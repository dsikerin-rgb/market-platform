### 1.1. Карта рынка и ревизия торговых мест

**Статус:** в работе, базовый контур есть.

Что уже есть:

- страница результатов ревизии;
- вкладки:
  - `Ревизионные решения`;
  - `Связь не подтверждена`;
  - `Дубли арендаторов`;
  - `Применено`;

- довести вкладку `Связь не подтверждена` до рабочего процесса:
  - подтвердить связь;

  - вернуть ошибочно скрытую связь в проверку.

### 1.2. Дубли арендаторов

**Статус:** рабочий MVP закрыт на prod.

Результат:

- дубли арендаторов выводятся во вкладке `Дубли арендаторов` человеческим языком;
- технические детали 1С скрыты в раскрываемом блоке `Технические данные 1С`;
- `Score`, `external_id`, `one_c_uid`, `dry-run`, `preflight`, `tenants:merge` убраны из основного пользовательского UI;
- есть два основных сценария:
  - если это один арендатор — `Подготовить слияние`;
  - если это разные арендаторы — `Это разные арендаторы`;
- скрытые пары видны в блоке `Скрытые пары`;
- ошибочно скрытую пару можно вернуть кнопкой `Вернуть в список`;
- после слияния дубль исчезает из активного списка;
- после нового импорта 1С проверено, что ранее слитая пара `#299 → #390` не вернулась как активный дубль;
- пользователь разобрал активные дубли на prod.

Основные PR/коммиты:

- `1b33761` — UI-сценарий слияния дублей из админки;
- `f50786b`, `e5a2861`, `fc4f11d` — уточнение текста подтверждений и автообновление списка;
- `763467e` — приведение экрана дублей к человеческому UX;
- `56f4c3a` — упрощение модалки слияния дубля;
- `5af9462` — возможность скрыть ложный дубль (`Это разные арендаторы`), миграция `tenant_duplicate_ignores`;
- `cb405a1` — блок `Скрытые пары` и `Вернуть в список`.

Таблицы/данные:

- `tenant_external_aliases` — защищает будущие импорты 1С от повторного создания уже слитого арендатора;
- `tenant_duplicate_ignores` — хранит пары, вручную отмеченные как разные арендаторы.

Проверочный сценарий после каждого импорта 1С:

1. Открыть `Результаты ревизии → Дубли арендаторов`.
2. Проверить, что ранее слитые пары не вернулись в активный список.
3. Проверить, что скрытые пары остались только в `Скрытые пары`.
4. Если появились новые активные пары — разобрать их по одной:
   - реальный дубль → `Подготовить слияние`;
   - разные арендаторы → `Это разные арендаторы`.

Отложенные улучшения:

- история слияний на карточке арендатора;
- краткая сводка по каждой карточке дубля: договоры, начисления, торговые места, пользователи;
- красивое форматирование даты в `Скрытые пары`.

### 1.3. Слияние арендаторов

**Статус:** рабочий сценарий есть, UI доведён до понятного MVP.

Что сделано:

- super-admin может подготовить слияние из UI;
- перед слиянием выполняется проверка, что будет перенесено;
- пользователь видит понятные роли:
  - `Основная карточка`;
  - `Карточка-дубль`;
- пользователь может поменять основную карточку;
- после проверки показывается блок `Будет перенесено`;
- перед фактическим слиянием есть подтверждение;
- перед слиянием система создаёт backup;
- если backup не создан — слияние не должно начинаться;
- после слияния список дублей обновляется автоматически;
- будущий импорт 1С использует alias и не создаёт повторно уже слитого арендатора.

Важно:

- backend-команда `tenants:merge` остаётся базовым безопасным механизмом;
- UI скрывает технические детали от пользователя, но не отменяет внутренние проверки;
- prod/staging миграции и любые DB-изменения выполняются только через контроль-барьер и backup.

### 1.4. Импорт 1С

**Статус:** рабочий импорт есть, нужен регулярный контроль качества после загрузок.

Что важно контролировать после каждого импорта:

- не восстановились ли уже слитые арендаторы отдельными карточками;
- не появились ли новые дубли арендаторов;
- не появились ли новые неподтверждённые связи торговых мест;
- не ухудшились ли статусы карты;
- корректно ли сработали alias из `tenant_external_aliases`.

Ближайший контроль:

- после следующего импорта открыть `Дубли арендаторов` и проверить активные пары;
- затем перейти к вкладке `Связь не подтверждена`.

## 3. Ближайшие задачи по приоритету

### P0 — контроль после следующего импорта 1С

Цель: убедиться, что очищенное состояние не ломается новым импортом.

Проверить:

- активные дубли не вернулись;
- скрытые пары остались скрытыми;
- слитые арендаторы не появились снова отдельными карточками;
- новые пары, если появились, действительно новые.

Результат проверки фиксировать в `docs/ops-log.md`.

---

### P1 — вкладка `Связь не подтверждена`

Цель: сделать такой же рабочий цикл, как для дублей арендаторов.

Текущая проблема:

- вкладка показывает спорные связи, но пользователю нужен понятный сценарий закрытия проблемы.

Ожидаемый сценарий:

1. Пользователь видит место и предполагаемую связь.
2. Если связь верная — подтверждает.
3. Если связь неверная — отклоняет/скрывает.
4. Если ошибся — возвращает в проверку.

Предварительные задачи:

- посмотреть текущий UI карточек `Связь не подтверждена`;
- определить, каких данных не хватает для решения;
- спроектировать минимальные действия без перегруза.

---

### P2 — сводка по карточкам дублей

Цель: уменьшить ручные переходы в карточки арендаторов.

Идея:

- на карточке дубля показывать краткую сводку:
  - договоры;
  - начисления;
  - торговые места;
  - пользователи.

Польза:

- проще выбрать основную карточку;
- меньше риск слияния не туда;
- быстрее ручная проверка.

Статус: отложено, не блокирует текущий процесс.

---

### P3 — история слияний арендаторов

Цель: дать пользовательский аудит действий.

Идея:

- на карточке арендатора показывать блок `История слияний`:
  - кто был слит;
  - в кого слит;
  - кто выполнил;
  - когда;
  - был ли создан backup.

Статус: отложено. Сейчас есть технические следы: лог, backup, alias, но нет удобного пользовательского отображения.

## 4. Правила deploy и БД

### 4.1. Общие правила

Запрещено на staging/prod:

- `php artisan migrate:fresh`;
- `php artisan migrate:refresh`;
- `php artisan db:wipe`;
- любые варианты `fresh`, `refresh`, `wipe`;
- destructive-команды с `--seed`.

Разрешённая миграция на staging/prod:

```bash
php artisan migrate --force
```

Только после контроль-барьера и backup.

### 4.2. Контроль-барьер перед миграциями/DB-экспериментами

Перед любым действием, которое может тронуть БД на staging/prod:

1. Показать Laravel DB config без секретов:
   - connection;
   - driver;
   - host;
   - port;
   - database;
   - username.
2. Показать размер PostgreSQL-БД.
3. Сделать backup через `pg_dump`.
4. Убедиться, что backup-файл существует и размер больше нуля.
5. Только потом запускать `php artisan migrate --force`.

### 4.3. Последние важные backup-файлы

Staging перед `tenant_duplicate_ignores`:

```text
storage/backups/market_staging_before_tenant_duplicate_ignores_20260524_093029.dump
```

Prod перед `tenant_duplicate_ignores`:

```text
storage/backups/market_prod_before_tenant_duplicate_ignores_20260524_094029.dump
```
