# Модель групп мест, dashboard-статистики и будущих площадей

См. также: [Проявленность временных group-карточек](market-space-period-effectiveness.md) — нормативное правило по проявленности временных group-карточек, физическому и учётному фонду, а также плану перехода от computed effectiveness к explicit group episodes.

Статус: продуктово-техническое правило для групп мест, статистики главной страницы, карты и будущей модели совместных мест.

## 1. Основные понятия

В системе нельзя смешивать три разных слоя:

1. Учётная единица.
2. Физический сегмент карты.
3. Совместно используемое место.

## 2. Учётная единица

Учётная единица — это объект, который видят 1С, таблица директора, договоры, начисления, долги и будущая статистика по площади.

К учётным единицам относятся:

- обычные места: `space_group_role = none` или пусто;
- parent-группы: `space_group_role = parent`;
- legacy child без `space_group_parent_id`, пока они не исправлены.

Child-место с parent не является самостоятельной учётной единицей.

## 3. Физический сегмент карты

Физический сегмент карты — это геометрическая часть торгового пространства.

Для grouped spaces физические сегменты представлены child-местами:

- `space_group_role = child`;
- `space_group_parent_id` указывает на parent-группу;
- `space_group_slot` определяет номер сегмента внутри группы.

Child нужен карте и ревизии, но не должен становиться отдельным договорным/финансовым объектом.

## 4. Parent-группа

Parent-группа — это учётная единица для 1С, таблицы директора, договоров, начислений и задолженности.

Parent-группа не должна иметь обычную активную торговую фигуру на карте.

Физически группа должна отображаться через child-фигуры. Если позже нужен общий контур группы, это должен быть отдельный overlay/outline, а не обычная shape-фигура торгового места, привязанная к parent.

## 5. Наследование child

Child с parent обязан наследовать бизнес-контекст от parent-группы:

- арендатора;
- занятость;
- договорный контекст;
- финансовый статус;
- цвет задолженности на карте.

Собственный `tenant_id` у child с parent — это data-quality warning, а не отдельная бизнес-истина.

## 6. Dashboard-статистика

Главная страница сейчас показывает количество мест, но стратегически более правильной метрикой будет площадь.

Так как площади приходят из 1С и таблицы директора только по обычным местам и parent-группам, dashboard-статистика должна опираться на учётные единицы.

### Считать в управленческой статистике

- обычные места;
- parent-группы;
- legacy child без parent до исправления данных.

### Не считать в управленческой статистике

- child-места с parent.

Причина: child-места являются физическими сегментами карты, но не имеют самостоятельной площади и не фигурируют в 1С/таблице как отдельные учётные строки.

Рекомендуемая трактовка карточек:

- `Учётных мест` = обычные места + parent-группы + orphan-child legacy;
- `Занято учётных мест` = занятые учётные единицы;
- `Свободно учётных мест` = свободные учётные единицы;
- `Заполняемость` = занятые учётные единицы / все учётные единицы.

Отдельная будущая метрика может показывать физические сегменты карты, но это не должно смешиваться с управленческой статистикой.

## 7. Будущая метрика площади

Когда заказчик поставит задачу считать площадь, правильная модель такая:

- площадь ordinary берётся из ordinary;
- площадь parent-группы берётся из parent;
- площади child не суммируются, потому что сейчас нет надёжных данных по каждому child;
- child-площади нельзя додумывать из геометрии или названия parent-группы.

## 8. Окраска карты задолженности

Цвет карты должен идти от договорного/финансового владельца.

### Обычное место

Цвет = статус задолженности обычного места.

### Parent-группа

Цвет = статус задолженности parent-группы.

### Child-место

Цвет = унаследованный статус parent-группы.

Popup child должен явно показывать, что арендатор и финансовый статус наследуются от группы.

## 9. Совместно используемое место

Совместно используемое место — это одно физическое/учётное место, которое одновременно используют несколько арендаторов или плательщиков.

Пример: `Склад 21`, где один номер отдела имеет несколько tenant/payment строк.

Это не parent/child-группа. Для такого кейса нужна отдельная будущая модель участников/долей.

Возможная будущая таблица:

`market_space_tenant_shares` или `market_space_participants`

Поля-кандидаты:

- `market_id`;
- `market_space_id`;
- `tenant_id`;
- `area_sqm` nullable;
- `share_percent` nullable;
- `period_start` nullable;
- `period_end` nullable;
- `source` = `1c`, `director_sheet`, `manual`;
- timestamps.

Окраска совместного места в будущем: агрегированный статус по участникам, по умолчанию худший статус выигрывает.

## 11. Что не входит в первый маленький fix

Не делать в первом PR:

- миграции;
- изменение 1С-связей;
- перенос договоров;
- удаление parent-shapes;
- создание shared-space таблиц;
- изменение import/debt/reconcile логики;
- пересчёт производственных данных.
