# Финансовый сигнал в ревизии мест

## Контекст

Начисление из 1С считается сильным финансовым фактом. Чтобы оно появилось в 1С, уже должен существовать контрагент и должна быть проведена финансовая операция. Поэтому локальная система не должна относиться к такому начислению как к слабому или шумному сигналу.

Начисление нельзя блокировать на входе. Его нужно сохранить, а затем проверить, собрана ли вокруг него рабочая связка:

- арендатор в локальной БД;
- договор;
- торговое место;
- текущий арендатор места.

## Целевая модель

1. Начисление всегда принимается как финансовый факт.
2. Новый контрагент из 1С должен попадать в локальную БД автоматически, если в импорте есть надежный внешний идентификатор.
3. Если арендатор не может быть создан или сопоставлен, система должна создать инцидент справочника.
4. Если начисление сохранено без договора, система должна создать инцидент договорного контура.
5. Если начисление по месту указывает на другого арендатора, чем карточка места, это должно попадать в `Результаты ревизии`.

## Что попадает в результаты ревизии

В `Результаты ревизии` попадают не все начисления без договора, а только те, которые меняют понимание состояния места.

Кейс должен попадать в ревизию, если выполняется одно из условий:

- начисление пришло на арендатора, отличного от текущего арендатора места;
- начисление пришло по месту, которое в карточке считается свободным;
- начисление пришло по месту, но локальный договор не найден;
- начисление пришло на нового арендатора, которого локальная БД не смогла создать или сопоставить.

Обычное начисление без договора, которое не меняет состояние места, остается в разделе `Начисления` и вкладке `Без договора`.

## Автоматическое создание арендатора

Для импорта начислений из 1С правильное поведение такое:

- если есть `tenant_external_id`, система должна найти или создать `tenant`;
- если найден локальный арендатор с тем же внешним идентификатором, используется он;
- если локального арендатора нет, создается новый активный арендатор в рамках рынка;
- если создать или сопоставить арендатора нельзя, начисление не должно молча теряться: нужен явный инцидент.

Слепое создание арендатора только по имени допустимо только как ручное действие оператора. Автоматический путь должен опираться на внешний идентификатор из 1С.

## Первый этап реализации

1. Добавить детектор финансовых сигналов для `MapReviewResultsService`.
2. Находить начисления с `tenant_contract_id = null`, у которых есть `market_space_id`.
3. Сравнивать арендатора начисления с текущим арендатором места.
4. Если они различаются, добавлять системную карточку в `Результаты ревизии`.
5. В карточке показывать источник: `Финконтур сообщает нового арендатора`.
6. Показывать арендатора из начислений, текущего арендатора места, период и статус договора.

## Второй этап реализации

1. Добавить отдельный тип инцидента для случаев, когда 1С прислала контрагента, но локальный `tenant` не создан.
2. Добавить действие оператора: `Создать/сопоставить арендатора`.
3. После сопоставления разрешить обычный сценарий `Сменить арендатора`.
4. Оставить договорный инцидент открытым, пока договор не будет найден, импортирован или привязан.

## Правило для кейса П56/2

Если по месту `П56/2` есть начисления на `Детятева О.С. ИП`, а карточка места указывает другого арендатора и договор не найден, это должно отображаться в `Результаты ревизии` как конфликт финансового контура с карточкой места.

Ожидаемый смысл карточки:

`По месту П56/2 есть начисления на Детятева О.С. ИП, но договор не найден, а карточка места связана с другим арендатором. Требуется подтвердить смену арендатора и отдельно разобрать договор.`
